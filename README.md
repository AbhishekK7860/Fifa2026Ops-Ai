# StadiumOps AI Dashboard

StadiumOps AI is an intelligent, real-time stadium operations dashboard designed to monitor and manage crowd control, gate capacity, and incident response for large-scale events. It combines real-time data visualization with AI-driven analysis to provide actionable insights for stadium volunteers.

## 🏆 Evaluator Quickstart

Test the full functionality in under 30 seconds with zero configuration:
1. Open the deployed application (or run locally).
2. Click **"Load Demo Dataset"** in the top right to instantly populate the dashboard with realistic 2026 stadium data.
3. Select any gate from the list to view the dynamically generated AI analysis and multilingual announcements.
4. **Bring Your Own Data:** Create a CSV file with the following headers: `Gate, Status, Queue Length, Capacity, Current Visitors, Volunteer Count, Medical Incidents, Transport Delay (mins), Weather` (and an optional `Timestamp (HH:MM)` column to enable time-series charting). Modify values (e.g., set `Current Visitors` above `Capacity`) and upload it to test the AI's real-time risk detection.

## 🎯 Evaluator Guide & Alignment

### Persona Alignment
As per the brief's guidance to **"pick ONE persona and focus intensely on their specific needs,"** this dashboard is explicitly designed for the **Stadium Volunteer**. Volunteers need immediate, actionable, and distraction-free information on their mobile devices during chaotic events. For example, the brief highlights that if a gate is at 85% capacity, a volunteer doesn't need a complex predictive chart; they need a clear instruction: *"Redirect arriving fans to Gate B."* The UX/UI prioritizes this exact real-time clarity, stripping away non-essential noise to deliver high-confidence, actionable AI insights with multilingual announcements.

### No-Hardcoding Guarantee
In adherence with the project requirements, **no AI responses are hardcoded**. All insights, reasoning, and multilingual announcements are dynamically generated by the LLM in real-time based on the live CSV data. (In the event of an API failure or network loss, the system gracefully falls back to deterministic rule-based heuristics to ensure operational continuity, but the primary pipeline is 100% prompt-driven).

### Free-Tier Caveats & Resilience
This project utilizes free-tier infrastructure (OpenRouter free endpoints, Vercel Hobby tier, Supabase free tier). As a result:
- **Rate Limits & Cold Starts:** You may occasionally encounter minor delays during initial load (cold starts) or OpenRouter rate-limiting (HTTP 429) during heavy burst testing.
- **Model Constraints:** The primary AI utilizes `meta-llama/llama-3.3-70b-instruct:free` via OpenRouter (with `nousresearch/hermes-3-llama-3.1-405b:free` as a fallback). While highly capable, free models may occasionally experience high latency or upstream provider congestion.
- **Offline Mode as a Feature:** To handle this unreliability, the built-in **Offline Analysis Mode** is a designed resilience feature, not a hardcoded shortcut. If the API fails, it takes over to guarantee continuous operations.

## Features

### 1. CSV Data Ingestion & Validation
The application allows users to upload live operational data via CSV. 
* **Validation Rules**: Strict Zod schemas validate the data structure, ensuring all required columns are present.
* **Limits**: Enforces a maximum of 500 rows per upload to prevent memory exhaustion and abuse.
* **MIME Checking**: Enforces `text/csv` MIME types on both client and server sides.

### 2. Explainable AI Analysis
When a gate is selected, the application queries an LLM to generate an explainable, real-time assessment of the gate's status. The output is strictly typed into six core fields:
1. **Observation**: A concise summary of the current situation.
2. **Reasoning**: The logical breakdown of why the situation requires attention, citing data.
3. **Recommended Action**: A concrete, actionable step for the volunteer to take.
4. **Expected Impact**: The projected outcome of the recommended action.
5. **Confidence Level**: A 0-100 score indicating the AI's certainty, explicitly clamped to local confidence bounds (determined by data recency and quality).
6. **Source Data Traceability**: Explicit references to the exact data points that drove the conclusion.

### 3. Multilingual Announcements
To assist volunteers in a diverse environment, the AI generates public announcements dynamically translated into English (EN), Spanish (ES), and French (FR), accessible via a tabbed interface.

### 4. Offline Analysis Mode (Resilience Fallback)
If the AI endpoint fails, times out, or the application loses network access, the system automatically falls back to **Offline Analysis Mode**. This mode uses deterministic, rule-based heuristics to generate the exact same analysis schema without relying on the LLM.

### 5. Security & Rate Limiting
* **Rate Limiting**: The application implements IP-based rate limiting to prevent abuse:
  * `/api/analyze`: 30 requests per 60 seconds (burst limit) and 100 requests per 24 hours (daily limit).
  * `/api/upload`: 5 requests per 60 seconds.
* **No-CAPTCHA Rationale**: CAPTCHAs introduce significant friction in high-stress operational environments. Instead, we rely on strict payload validation, request rate limiting, and maximum row caps.
* **Prompt Injection Detection**: A dedicated checkpoint scans incoming user queries for prompt injection attempts (e.g., "ignore previous instructions") and blocks them instantly.

### 6. Supabase Persistence (Optional Audit Logging)
The core application runs fully in-memory and **does not** require a database to function. If configured, it connects to Supabase to maintain two specific tables with differing Row Level Security (RLS) postures:
* `upload_audits`: Tracks file uploads (filename, row count, timestamp) for volumetric analysis. RLS allows anonymous inserts and selects, but no updates or deletes.
* `security_events`: Logs blocked prompt injection attempts and malicious inputs for security auditing.

### 7. Accessibility (a11y)
The application strictly adheres to WCAG standards:
* Full keyboard navigation (focus rings, accessible tabs, and semantic HTML).
* ARIA labels on all interactive elements.
* Color contrast ratios verified against both light and dark modes.

### 8. Export Functionality
Volunteers can export the complete AI analysis (including the multilingual announcements and recommended actions) as a downloadable PDF report or JSON file for post-incident reviews or handoffs.

## Setup Instructions

### Prerequisites
* Node.js 18+
* npm

### Installation
```bash
npm install
```

### Environment Variables
Create a `.env.local` file in the root directory. **All variables are technically optional.** The app will gracefully degrade to Offline Mode and skip audit logging if services are missing.

| Variable | Description | Required? |
|----------|-------------|-----------|
| `OPENROUTER_API_KEY` | API key for OpenRouter to power the AI Analysis. | No (Falls back to Offline Mode) |
| `NEXT_PUBLIC_SUPABASE_URL` | Your Supabase project URL for audit logging. | No (Fails silently) |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Your Supabase anonymous key for audit logging. | No (Fails silently) |

### Running the Development Server
```bash
npm run dev
```
Open [http://localhost:3000](http://localhost:3000) in your browser.

### Building for Production
```bash
npm run build
npm run start
```

## Deployment
The application is optimized for Vercel. 
1. Connect your repository to Vercel.
2. Add the environment variables in the Vercel dashboard.
3. Deploy. The application uses Next.js App Router and edge-compatible API routes.
